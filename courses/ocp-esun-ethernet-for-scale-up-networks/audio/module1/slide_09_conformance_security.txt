The specification splits requirements between endpoints and switches. For endpoints — the accelerator-side device — PFC support is required. CBFC and LLR are optional. If they are supported, they must be configurable on a per-port basis. The endpoint must produce the "ee-sun" Header, setting EH-Co S, the Flow Label, and T T L correctly, and it must announce E C N capability and respond to E C N marks from the fabric. For switches, the bar is higher. PFC is required, CBFC is required, and LLR is required — all three configurable per port. Forwarding is destination-address based on the forty-eight-bit Ethernet D A. Because there's no I P header and no learning protocol, those addresses are configured statically at switch bring-up — there is no Mac learning, no aging, and the spec is explicit that switches must not broadcast "ee-sun" packets to unknown destinations. Class of service from EH-Co S takes precedence over class of service in any eight-oh-two dot one Q tag or in CBFC. The Flow Label is used for hashing whenever the F bit is set. Switches indicate congestion via E C N when appropriate, and they decrement T T L on every hop. Security follows directly from the design choice. Because the "ee-sun" Header replaces the I P header, "ee-sun" is not compatible with I P sec. The recommended path for confidentiality, integrity, and authentication is MACsec, which operates at Ethernet layer two as point-to-point link encryption. End-to-end encryption is out of scope for "ee-sun" itself; that's a transport concern. The summary table in the spec maps each feature to required versus optional cleanly, and a vendor can self-certify against it. If you build to the table, you are "ee-sun" one point zero conformant.
