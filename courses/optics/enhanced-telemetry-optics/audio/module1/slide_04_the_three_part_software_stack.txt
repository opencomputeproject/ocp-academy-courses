# pronunciation: CMIS -> C M I S; CDB -> C D B; I2C -> I squared C; SONiC -> sonic; PMD -> P M D; VDM -> V D M; DDMI -> D D M I; SNR -> S N R; PAM4 -> PAM four; MPI -> M P I; FEC -> feck; BER -> B E R; LOS -> L O S; LOL -> L O L; SDK -> S D K; API -> A P I
The source document names three components in a compliant software system. The first is the transceiver itself: firmware, telemetry metrics, event logging, and the command behavior needed to expose those values.

The second is host software, typically on the switch. This is the layer that talks to the local module over I squared C, asks for local or remote partner data, and turns module responses into tools an operator can use.

The third is telemetry-server software. The spec expects telemetry to be compiled, summarized, and reported at the fabric level, but it does not prescribe how every operator builds that server side. That boundary is important. The course teaches the common module-to-host interface, because that is the part the contribution standardizes.
That layered view also helps with responsibility. The transceiver owns what it can measure and preserve. The host owns how it asks and presents. The telemetry server owns broader correlation across the fleet.
