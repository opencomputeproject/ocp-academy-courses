# pronunciation: CMIS -> C M I S; CDB -> C D B; I2C -> I squared C; SONiC -> sonic; PMD -> P M D; VDM -> V D M; DDMI -> D D M I; SNR -> S N R; PAM4 -> PAM four; MPI -> M P I; FEC -> feck; BER -> B E R; LOS -> L O S; LOL -> L O L; SDK -> S D K; API -> A P I
This course is based on an OCP Community contribution written by volunteers from Credo and Oracle. The course has four parts. First, we frame the visibility gap that shows up when optical links span switch ports, transceiver modules, fiber paths, and remote endpoints that may sit inside bare-metal servers.

Second, we work through the categories of telemetry that make link state actionable: optical quality, electrical quality, errors, recovery behavior, temperature, power, and administrative context.

Third, we look at the command-reply workflows that move this information between host software and modules. That includes C D B, event logs, and remote C M I S access.

Finally, we connect the telemetry model to firmware management, host APIs, and deployment readiness. By the end, you should be able to explain the operating model without getting trapped in byte maps.
Keep one question in mind while you move through the modules: would this information help an operator decide the next action faster? If the answer is yes, it belongs in the reliability workflow.
