# pronunciation: CMIS -> C M I S; CDB -> C D B; I2C -> I squared C; SONiC -> sonic; PMD -> P M D; VDM -> V D M; DDMI -> D D M I; SNR -> S N R; PAM4 -> PAM four; MPI -> M P I; FEC -> feck; BER -> B E R; LOS -> L O S; LOL -> L O L; SDK -> S D K; API -> A P I
The overview gives us six functions to keep in mind. First, compliant transceivers monitor links in mission mode, meaning the module collects statistics while the link is doing useful work.

Second, the module reports link health through a common metric and tracks telemetry over time. Third, the module supports transparent in-band communication so the two ends of the link can identify each other and exchange data.

Fourth, the module can notify and log events in non-volatile memory. Fifth, it supports hitless firmware update on local and remote transceivers. Sixth, switch software can harvest module telemetry and prepare it for streaming to a server.

Those functions are easiest to remember as a chain: measure, score, reach the far end, preserve the story, update safely, and feed the operational system.
Notice that none of these functions stands alone. Monitoring without logging loses history. Logging without remote reach loses endpoint context. Firmware update without status feedback increases risk. Reliability comes from the functions working together.
