# pronunciation: CMIS -> C M I S; CDB -> C D B; I2C -> I squared C; SONiC -> sonic; PMD -> P M D; VDM -> V D M; DDMI -> D D M I; SNR -> S N R; PAM4 -> PAM four; MPI -> M P I; FEC -> feck; BER -> B E R; LOS -> L O S; LOL -> L O L; SDK -> S D K; API -> A P I
Before we talk about health scores, the module needs baseline status. The spec points to common C M I S Bank zero pages, configuration and status pages, V D M pages, C D B command pages, C D B payload pages, and custom vendor pages.

For this course, the important point is not the exact page number for every field. The important point is that enhanced telemetry sits on top of a structured management interface. The host can retrieve standard identity and status information, then use the enhanced telemetry definitions to ask deeper reliability questions.

That layering keeps the course practical. You should know where the detailed maps live, but your first mental model should be: identity and state first, then measurements, then health interpretation.
Baseline pages are also how software avoids misreading context. Identity, capability, and state information should be checked before a tool assumes that a telemetry field is available, current, or meaningful for the module under test.
