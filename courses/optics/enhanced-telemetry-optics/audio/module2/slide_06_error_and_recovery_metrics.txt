# pronunciation: CMIS -> C M I S; CDB -> C D B; I2C -> I squared C; SONiC -> sonic; PMD -> P M D; VDM -> V D M; DDMI -> D D M I; SNR -> S N R; PAM4 -> PAM four; MPI -> M P I; FEC -> feck; BER -> B E R; LOS -> L O S; LOL -> L O L; SDK -> S D K; API -> A P I
Quality metrics tell you about margin. Error and recovery metrics tell you how that margin is affecting service. The spec includes pre-feck B E R, feck bin histograms, uncorrectable codewords, relink counts, loss of signal, loss of link, and uptime since the last new link-up.

These values help distinguish a noisy but corrected link from a link that is actually dropping and reacquiring. That distinction matters during maintenance. A corrected error pattern may call for trend monitoring or qualification attention. Repeated relinks or uncorrectable codewords may call for immediate isolation.

The strongest workflows combine both views. You want the signal-quality evidence and the service-impact evidence together, because that combination points to severity and likely location.
The time dimension is just as important as the count. A single corrected error burst, a rising histogram trend, and repeated relinks across a short window are different operational stories even if they all appear under error metrics.
