# pronunciation: CMIS -> C M I S; CDB -> C D B; I2C -> I squared C; SONiC -> sonic; PMD -> P M D; VDM -> V D M; DDMI -> D D M I; SNR -> S N R; PAM4 -> PAM four; MPI -> M P I; FEC -> feck; BER -> B E R; LOS -> L O S; LOL -> L O L; SDK -> S D K; API -> A P I
This module turns telemetry into workflow. C D B gives host software a repeatable command-reply sequence: prepare, trigger, poll, and retrieve. Page 9Fh and payload pages provide the structure for that exchange.

Status checking is the guardrail. A tool should not turn an unfinished or failed command into misleading output.

Event logs then add history. They preserve sequence, time, event type, and detail so operators can reconstruct what happened after the link changed state. Remote C M I S access completes the visibility path by reaching the far endpoint through the local module.

Together, these workflows make telemetry operational instead of just measurable.
The main risk in this layer is false confidence. A command that appears to return data is only useful if the software followed the workflow and checked status. Evidence needs process behind it.
