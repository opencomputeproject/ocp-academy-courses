# pronunciation: M-PNP -> "M P N P"; HPM -> "H P M"; BMC -> "B M C"; Fru -> "Fru" (read as the word, not letters)

The vision is a three-rung ladder. The bottom rung is where we are: Plug-and-Code. Everything custom, every HPM a separate enabling effort. The top rung is full Plug-and-Play. The BMC boots without prior knowledge of the HPM. It reaches across the connector, retrieves a set of self-describing definition files the HPM vendor wrote at design time, and from those files alone it knows where every sensor lives, what every register does, which buses route where, and how to bring the platform up. Runtime loading of all definitions; baseline functionality for any conforming HPM. Between those two rungs sits Plug-and-Play Lite. Some files are dynamically loaded, others still compiled in. That's roughly where the version zero point five specifications land. Two things make the vision possible. First, the hardware vendor knows the hardware better than anyone, so the vendor writes the descriptive files and ships them with the board. Second, the file formats are defined once by OCP and consumed by every conforming BMC. The result is what the workstream calls commoditizing the non-value-added platform enablement work — the Fru plumbing, the discovery code, the sensor wiring — so engineering effort goes back to the parts customers actually pay for.
