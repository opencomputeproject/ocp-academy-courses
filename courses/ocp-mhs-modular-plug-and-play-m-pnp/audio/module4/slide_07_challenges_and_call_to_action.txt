# pronunciation: HPM -> "H P M"; M-PNP -> "M P N P"; JSON -> "Jason"; LED -> "L E D"; PCIe -> "P C I E"; CEM -> "C E M"; FPGA -> "F P G A"; BMC -> "B M C"; Bios -> "BYE-oss" (mixed case so it reads as a word); DC-MHS -> "D C M H S"; frupack.py -> "Fru pack dot P Y"

Three open challenges the workstream is wrestling with. One: platform and chassis-level self-descriptive JSON. The HPM knows what's on the HPM, but it doesn't know what chassis it's in — and chassis-level decisions like LED strategy and thermal policy need to live somewhere. The workstream is still figuring out where and how to define a platform-level JSON. Two: PCIe add-in cards. OCP NIC and PCIe CEM already define a lot, but the workstream is finding gaps that need to be filled — and may need to be filled in those existing specs rather than in M-PNP. Three: prototyping. Specifications without working code are aspirational. The workstream needs a full prototype platform — FPGA, BMC, Bios — to validate everything. Call to action: review the specs, contribute via the DC-MHS public mailing list, watch the two Tech Talks linked in the bundle, build a JSON file for hardware you care about using frupack.py, and bring questions to the OCP Innovation Village Station the next time you're at a summit. The community needs hands on hardware.
