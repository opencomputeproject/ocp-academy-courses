# pronunciation: HPM -> "H P M"; FPGA -> "F P G A"; BMC -> "B M C"; MHS -> "M H S"; SoC -> "S O C"; CSR -> "C S R"; CSRs -> "C S Rs"; JSON -> "Jason"; API -> "A P I"; LUTs -> "L U Ts"

The HPM FPGA standardization concept looks like a layered cake. In the middle sits a set of MHS-standardized building blocks — open-source-able IP — with a stable interface above and a stable interface below. Above the standardized layer is HPM vendor customization: vendor-specific power sequencing logic, the SoC-specific reset choreography, whatever the vendor adds to differentiate their HPM. Below the standardized layer is the system integrator's own custom logic — additional features the integrator wants without breaking baseline interoperability. The goal is to make the middle layer rich enough that the BMC sees a uniform, predictable FPGA across every HPM vendor, while leaving the top and bottom layers open enough that vendors can still compete on the parts that matter. The specification doesn't dictate FPGA resource consumption — no maximum LUTs, no required memory size — and it doesn't dictate how a vendor implements their power sequencing state machine internally. What it dictates is the API: the CSRs, the memory map, the virtual wires, and the JSON description of all of it. The result is open-source-able IP for the standardized blocks, with a clean handoff to vendor and integrator logic.
