# pronunciation: HPM -> "H P M"; FPGA -> "F P G A"; BMC -> "B M C"; LTPI -> "L T P I"; SGPIO -> "S G P I O"; M-PESTI -> "M PESTI"; M-XIO -> "M X I O"; MXIO -> "M X I O"; M-PIC -> "M PIC"; PICPWR -> "P I C P W R"; SPI -> "S P I"; I2C -> "I two C"; I3C -> "I three C"; CSR -> "C S R"; CSRs -> "C S Rs"; JSON -> "Jason"; IP -> "I P"; IO -> "I O"; GPIOs -> "G P I Os"

The catalog defines thirteen IP blocks. Six are required. Power Sequencing and Management defines unified HPM power states and the CSRs the BMC writes to request transitions. System and Device Reset Control covers virtual reset for MXIO connectors, OCP slots, and on-board devices. The Virtual Wires Framework maps internal FPGA signals to the SGPIO or LTPI transport — we'll come back to it in slide eight. LTPI or SGPIO — at least one — provides the BMC-facing transport. M-PESTI support includes mapping CSRs into the FPGA memory map for peripheral identification. FlexIO MXIO and M-PIC PICPWR sideband configuration handles the configurable IO pins on the M-XIO connectors and M-PIC sideband. Seven blocks are optional but defined. SPI for random-access memory map. Generic Mux Control for I2C, I3C, and other system bus mux trees. LED Control for system LEDs and seven-segment displays. Cooling Control for fans and pumps. System Telemetry for telemetry interfaces. I2C and I3C blocks for the FPGA's own bus participation. And General Use GPIOs. There's also vendor extensibility. Vendor-defined IP blocks must be mapped into the F-P-G-A memory map, must use the virtual wire framework for any signals that need external transport, and must provide a self-describing JSON description with their CSRs. That last point is the punchline: even the parts of the FPGA the spec doesn't standardize are required to describe themselves in a format the BMC can read.
