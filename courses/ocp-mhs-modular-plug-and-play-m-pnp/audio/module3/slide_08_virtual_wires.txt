# pronunciation: FPGA -> "F P G A"; HPM -> "H P M"; BMC -> "B M C"; DC-SCM -> "D C S C M"; SGPIO -> "S G P I O"; LTPI -> "L T P I"; JSON -> "Jason"; GPIO -> "G P I O"; M-PNP -> "M P N P"

Virtual wires solve a physical problem with a logical layer. The HPM FPGA has thousands of internal signals — power-good indicators, reset requests, interrupt lines, status bits — that the BMC needs visibility into. You can't run a separate pin per signal across the DC-SCM connector. The virtual wires framework lets the spec define a list of required and optional logical signals, and then the FPGA maps each one onto a transport: a bit position in an SGPIO frame, or a position in an LTPI virtual wire channel. The mapping itself is described in JSON. The BMC, at discovery time, reads the FPGA's JSON, learns which transport carries which virtual wire, and from that point on can manipulate FPGA internal state without a dedicated pin per signal. The spec defines a baseline list of required virtual wires that every M-PNP-conforming FPGA must expose. Vendors can add more, and the JSON tells the BMC where to find them. The same framework also handles GPIO mapping for the FPGA's package pins. So the picture is: a list of logical signals, a mapping per signal to a physical transport bit, and a JSON description the BMC reads at boot — and the framework runs without any code changes in the BMC when a new HPM shows up.
