# pronunciation: FPGA -> "F P G A"; HPM -> "H P M"; BMC -> "B M C"; JSON -> "Jason"; CSRs -> "C S Rs"; Fru -> "Fru" (read as the word, not letters); DMTF -> "D M T F"

The fpga JSON schema, version zero point one point three, is what makes the standardization concept actually usable at runtime. The FPGA build process generates a JSON file describing exactly this build of this FPGA. It declares the general information — vendor, version, build hash. It declares every IP block present in the FPGA — required ones from the catalog plus vendor-defined ones — with each block's role and capabilities. It declares the IP block's CSRs: register name, offset in the memory map, access type, bit fields, reset values, the works. It declares the memory map sections by name and address range. And it declares every virtual wire mapping. The BMC retrieves this JSON during discovery — by following the File Descriptor in the HPM's DMTF Fru — and from that JSON alone it now knows the exact register layout, the exact virtual wire mapping, and the exact memory map of this specific FPGA build. No prior knowledge required. That's the whole point. Combined with the hardware-component JSON describing the HPM's physical topology, the BMC has everything it needs to drive the platform without a custom firmware image. The two JSONs together are the description; the FPGA is the execution surface.
