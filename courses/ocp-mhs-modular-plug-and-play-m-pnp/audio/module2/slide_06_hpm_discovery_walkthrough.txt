# pronunciation: HPM -> "H P M"; BMC -> "B M C"; FPGA -> "F P G A"; SCM -> "S C M"; Fru -> "Fru" (read as the word, not letters); SPDM -> "S P D M"; I2C -> "I two C"; JSON -> "Jason"; PLDM -> "P L D M"

Let's walk Phase one slowly because it sets the security model for everything that follows. The BMC, in pre-standby, reads the IPMI Fru from I2C bus zero, address 0xA0. It provisionally trusts that data — provisionally, because the trust mechanism hasn't run yet. If the Fru carries a digital certificate and signature, the BMC validates them now. Optionally, SPDM runs as a second attestation. Once the BMC verifies compatibility with the HPM, the electrical handshake begins. The SCM asserts SCM_HPM_STBY_EN; the BMC moves to the Standby Hold-Off power state. The HPM asserts HPM_SCM_STBY_RDY when its auxiliary rails are in regulation. At this point — and this is the security gate — the BMC can attest the HPM FPGA. The FPGA is powered but held in reset; nothing has run yet that the BMC hasn't approved. Once the FPGA is attested, the SCM de-asserts SCM_HPM_STBY_RST_N, the FPGA exits reset, and the system transitions to Standby Run. With the FPGA running, the BMC can now retrieve the local HPM JSON attributes — either via PLDM File Transfer if the device is reachable that way, or by following the File Descriptor table in the DMTF Fru. Then, with main standby in place, it retrieves the extended JSON. By the end of Phase one, the BMC has a complete, validated description of the HPM in memory.
