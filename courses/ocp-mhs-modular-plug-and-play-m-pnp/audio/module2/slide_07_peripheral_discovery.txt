# pronunciation: HPM -> "H P M"; BMC -> "B M C"; M-PESTI -> "M PESTI"; Fru -> "Fru" (read as the word, not letters); PDB -> "P D B"; PDBs -> "P D Bs"; UBM -> "U B M"; CEM -> "C E M"; JSON -> "Jason"; I2C -> "I two C"; I3C -> "I three C"; M-CRPS -> "M C R P S"

Phase two covers the non-endpoint peripherals — the intermediate boards. Risers, backplanes, power distribution boards, control panel boards. Each one exposes an Fru EEPROM, often a small microcontroller, microprocessor, or CPLD, reachable from the BMC over I2C, I3C, or M-PESTI. The mechanism mirrors the HPM: a local IPMI record for identification, an optional DMTF Fru for richer description, and a JSON file that the BMC retrieves to learn the topology under that peripheral. The bundle ships six concrete examples you can hold in your hand. A single-slot x8 PCIe CEM riser. A single-slot x16 riser. A two-slot x8 riser. A two-slot x16 riser. A UBM backplane for NVMe drives. And a Type 2 power distribution board. Each ships as both the raw JSON file and a PDF rendering — open the PDF first if you're new to the schema. Once Phase two completes, the BMC knows every PCIe lane in the system, every drive slot, every fan tach, every power rail. Phase three then enumerates the endpoints living in those slots — NICs that follow the OCP NIC three point zero specification, drives via standard out-of-band protocols, M-CRPS power supplies, OCP CEM accelerator cards. Phase three leans heavily on existing endpoint specifications; M-PNP focuses on getting you to the endpoint, not replacing what's already there.
