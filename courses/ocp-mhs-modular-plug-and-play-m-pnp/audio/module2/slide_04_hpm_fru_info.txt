# pronunciation: HPM -> "H P M"; Fru -> "Fru" (read as the word, not letters); BMC -> "B M C"; Bios -> "BYE-oss" (mixed case so it reads as a word); IPMI -> "I P M I"; DMTF -> "D M T F"; EEPROM -> "E E PROM"; FPGA -> "F P G A"; JSON -> "Jason"; I2C -> "I two C"; SPDM -> "S P D M"

HPM Fru information lives in three layers. The first layer is an IPMI-format multi-record on an EEPROM at a fixed location — I2C bus zero, address 0xA0 — and it must be operational in the pre-standby power state. That's important: it has to be readable before the SCM has even granted main standby power, because the SCM uses what it reads to decide whether to grant power at all. The IPMI record carries vendor ID, component class, board identification — the bare minimum for the BMC to recognize what's plugged in. The second layer is a DMTF DSP0220 Fru record, also stored locally. This adds richer structured fields and, optionally, a digital certificate and a digital signature so the BMC can authenticate the Fru data using standard mechanisms — or, optionally, run SPDM as an additional attestation step. The third layer is extended attributes — and these are JSON, not IPMI. They can be large, so they don't have to live on the local EEPROM. The DMTF Fru contains a Files Record with a File Descriptor table. Slot one points to the local JSON attributes, slot two to the extended JSON. The Resident bit tells the BMC whether the file is on the EEPROM itself or somewhere else; if it's elsewhere, the Context field's two high-order bits encode where — Resident bit clear with context value 11 binary means a virtual EEPROM emulated by the FPGA, 10 binary means a Bios firmware volume. The result: a clean, hierarchical, signable Fru surface the BMC can walk top-down.
